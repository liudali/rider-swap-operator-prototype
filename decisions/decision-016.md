| 项 | 内容 |
|----|------|
| 日期 | 2026-07-16 |
| 阶段 | MVP / 发布前详设 |
| 结论 | **推荐**：用「一条 C 端主脊柱 + 三条支线」串功能；评审/演示默认从**个人骑手购买套餐**讲起，其余菜单按「前置条件 / 换电中 / 换电后结算」挂到脊柱上。 |

## 决策问题

功能菜单过多，角色与模式并行，听众/研发容易「见森林不见路径」。如何把功能串成可讲、可验收、可排期的主链路？

## 调用的大师视角

- **俞军（价值公式）**：主价值是「骑手有电可跑」；购买套餐是权益入口，换电是履约。菜单价值 = 是否缩短「购权益 → 取电」路径，而非功能点数。
- **乔布斯（减法）**：演示与一期讲稿只保留脊柱；渠道四模式、融资、合伙人提现标为**支线开关**，不默认全讲。
- **王慧文（双边）**：平台双边是「骑手需求电」↔「运营商供网」；渠道是 B 端获客/额度的变体，不是第二条无关业务。

## 核心判断

| 层级 | 内容 |
|------|------|
| **脊柱（默认演示）** | 定价上架 → 骑手购套餐支付 → 清分 1% → 扫码换电 →（可选跨网）→ 运营商提现 |
| **支线 B** | 渠道获客/人天（与脊柱汇合于「换电」） |
| **支线 C** | 资产与融资（设备可换电的前置） |
| **支线 D** | 分润与提现（收入分配，不改变换电） |

## 反方最强论据

渠道人天、设备租赁在真实 GMV 可能更大，「从个人购套餐讲起」会低估 B 端。缓解：开场 30 秒声明「两条骑手路径」，个人路径讲透后再切渠道对照表。

## 失效条件

商务明确一期只做渠道、不做 C 端个人套餐；或评审对象只要治理后台、不关心骑手端。

## Non-goals

- 不要求原型一次跑通全部角色菜单
- 不把总览分析、信用档位当作脊柱节点
- 不另起「平台营销」为第三条主业务（仍挂在购套餐变体）
