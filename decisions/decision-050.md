# 决策卡片 050 · 骑手端押金环节（免押 / 实缴）

| 项 | 内容 |
|----|------|
| 日期 | 2026-07-21 |
| 阶段 | MVP / C 端补漏 |
| 结论 | **已锁定**：用户端必须可见押金环节；可选**微信支付分免押**或**实缴押金**。个人购套餐同笔办理；**渠道人天首次领取电池前**须完成押金环节 |

## 结论

1. **个人套餐（P04）**：购套餐结算页展示「套餐费 + 电池押金」；用户二选一：
   - **微信支付分免押**（达标则押金 ¥0，记「信用免押」）
   - **支付押金**（按运营商「押金设置」数额同笔实缴，记「实付在押」）
   - `[假设]` 芝麻信用作为并列免押通道，与支付分「任一路达标即可」一致（后台门槛已配置）
2. **渠道人天（C04）**：权益到账后、**首次领取电池前**须完成同一套押金/免押选择；未办结不可出电领取。
3. **台账类型**：人天池骑手办结后记 **实付在押 / 信用免押**，**不再**默认静默「渠道担保」。
4. **设备租赁白名单**：仍可走渠道信用/渠道担保（B 端），本期 C 端不强制首领押金 UI。`[待确认]` 若租赁也要对齐 C 端首领押金，另开决策。

## 对历史规则的修订

| 原文 | 修订 |
|------|------|
| D38「渠道用户默认免押，押金算在渠道」 | **人天池 C 端**改为骑手本人免押或实缴；渠道信用仍可作 B 端风控参考，但不替代骑手端押金环节 |
| decision-005 / 押金文档「渠道担保=人天池」 | 人天池 → 实付/免押；**渠道担保**主要保留给设备租赁等仍由渠道承担押金责任的场景 |

## 支持理由

- **俞军**：押金是持电池的真实成本感知；静默「渠道担保」使 C 端无替换成本意识，丢电追责无用户侧契约感。
- **乔布斯**：结算/首领各保留一次清晰二选一，不拆成多页「信用产品」。

## 反对理由

- 渠道侧可能认为已采购人天含担保，再向骑手收押增加摩擦与客诉。
- 支付分拉起失败时首领链路变长。

## 失效条件

- 监管要求渠道统缴、禁止向骑手个人收押；或支付分/芝麻接口长期不可用且无实缴兜底。
