# 决策卡片 009 · 平台「服务变更」归入用户管理

| 项 | 内容 |
|----|------|
| 日期 | 2026-07-15 |
| 阶段 | MVP / 信息架构调整 |
| 结论 | **已确认**：平台侧「服务变更」从 **订单管理** 挪至 **用户管理** 二级菜单（用户信息 / 用户押金统计 / **服务变更**）。订单管理保留：套餐购买 / 换电 / 渠道商订单。 |

## 为何改

- **乔布斯减法**：订单管理收敛为「成交单据」；冻结/解冻/中途完结更贴近用户服务状态，挂在用户管理更易找。
- **俞军**：巡检「某人有什么变更」的替换成本低于在订单三层里翻找；平台只读追溯场景与用户台账相邻。

## 规则

1. 平台 L2：**用户管理 → 服务变更**；筛选与列表能力不变（只读）。
2. 变更记录 / 审计跳转：服务变更类进 `platformUsers` + `serviceChange`；套餐/换电仍进订单管理。
3. 运营商侧「我的订单 · 服务变更」**不变**（本卡仅平台 IA）。

## 支持 / 反对 / 失效

| | 内容 |
|--|------|
| 支持 | 用户态与服务变更同屏上下文；订单管理更纯粹 |
| 反对 | 习惯「服务变更=订单」的运营需一次认知迁移 |
| 失效 | 服务变更升级为强审核/资金操作主入口，应独立一级或并入退款管理 |
