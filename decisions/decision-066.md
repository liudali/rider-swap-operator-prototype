## 决策问题

平台「用户管理 → 用户信息」列表中，押金类型与在押/退押语义混在同一列，客服扫读成本高；持有电池展示不统一。

## 调用的大师视角

- **俞军（价值公式）**：拆列后「有没有押 / 还在不在」一眼可读，替换成本低于教客服读复合标签。
- **乔布斯（减法）**：去掉「实付在押」等粘连标签；副行说明不再塞进持有电池列。

## 已确认规则

### 1. 电池押金（方式）

| 取值 | 展示 |
|------|------|
| 实付 | `实付（实收¥xx）` |
| 信用免押 | `信用免押（微信支付分 xx分）` 或 `信用免押（芝麻信用 xx分）` |
| 渠道担保 | `渠道担保（xxx渠道）` |
| 无记录 / 已退还 | `——` |

### 2. 押金状态（新列）

| 取值 | 含义 |
|------|------|
| 在押 | **仅实付**：款仍在运营商账上 |
| 退押中 | **仅实付**：已发起退押未结清 |
| —— | **非实付**（信用免押 / 渠道担保）或未使用服务或已退还 |

> **规则**：没有实付押金的用户，押金状态统一为「——」，不展示「在押」。

### 3. 持有电池

- 持有：`电池编码-SOC xx%-SOH yy%（归属 xxx运营商）`（**SOH 为百分比**；**必须有归属运营商**，台账缺失时回退换电单三元组 B / 柜机运营商）
- 否则：`未持有`（不附副行原因）

### 4. 筛选

- **电池押金**：全部 / 实付 / 信用免押 / 渠道担保 / ——
- **押金状态**：全部 / 在押 / 退押中 / ——

## 支持 / 反对 / 失效

| | |
|--|--|
| **支持** | 字段语义与「用户押金」明细、退款管理对齐；客服路径清晰 |
| **反对** | 多一列占宽；运营商「用户」页未同步拆列（本期仅平台） |
| **失效** | 若产品改回「类型+状态」单列复合标签 |
