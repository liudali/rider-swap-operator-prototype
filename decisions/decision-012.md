# 决策卡片 012 · 平台营销改为立减券 · 购时锁 OP · 不代收

| 项 | 内容 |
|----|------|
| 日期 | 2026-07-15 |
| 阶段 | MVP / 架构更正 |
| 结论 | **已确认**：平台统一营销以 **立减优惠券** 形式触达用户；**购买套餐时必须锁定运营商**；用户支付款进 **锁定运营商子商户**。**废止**「平台代收用户款再转运营」与「首服取电后拨付」。 |

## 为何改

- **第一性原理**：套餐服务由运营商履约，收款主体应是运营商，避免平台过账造成资金链/合规/对账复杂。
- **俞军**：旧架构「先付平台、待激活、再拨付」替换成本高（用户空窗、资金在途、待激活拦截）。立减券 + 实时锁 OP + 直付运营商，新体验更贴「买完就能用」。
- **贝佐斯（逆向）**：三年后仍希望与微信分账、运营商提现模型一致；券是营销层，不改变清分主干。

## 新规则（相对旧架构 C）

| 维度 | 旧（已废止） | 新 |
|------|-------------|-----|
| 优惠形态 | Campaign 活动价（价签改写） | **立减优惠券**（展示原价 − 券面额 = 实付） |
| 收款 | 平台子商户 | **锁定运营商**子商户 |
| `userOwner` | 首服取电 | **支付成功时**写入锁定 OP |
| 资金二次划转 | 激活后拨付（实付 − 营销服务费） | **无**用户款过桥；营销服务费改为协议结算 `[假设]` 月结应付 |
| 链接 | 不带 `op` | **必须带 `op=`**（或下单前强制选参与运营商） |
| 待激活 | 有 | **无**（支付即可服务） |

## 资金公式（单笔）`[假设]`

| 项 | 口径 |
|----|------|
| 原价 / 券面额 / 实付 | `list_price` / `coupon_amount` / `paid = list − coupon` |
| 用户支付 | 进 **锁定 OP** 子商户 |
| 1% 技术服务费 | `paid × 运营商 C 端费率`（支付分账，与直购同） |
| 券成本 | **已确认（013）**：**默认运营商承担**（面价差让利）；无平台补贴出账 |
| 营销服务费 | 按协议 **元/单** 向运营商计应付，**不**从用户过款中扣 |

## 支持 / 反对 / 失效

| | 内容 |
|--|------|
| 支持 | 与架构 B 清分一致；去掉代收合规负担；用户路径更短 |
| 反对 | 链接必须带 OP，削弱「一个品牌链全国投」的自由度；多 OP 投放要生成多链或强制选站 |
| 失效 | 监管强制平台统收；或品牌收银台必须进平台商户号 |
