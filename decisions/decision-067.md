# 决策卡片 067 · 套餐购买订单：押金方式 / 收款状态分列

| 项 | 内容 |
|----|------|
| 日期 | 2026-07-24 |
| 阶段 | 发布前 |
| 结论 | **已锁定**：运营商「套餐购买订单」列表将原「押金」列拆为 **押金方式** 与 **收款状态**；详情与筛选同步 |

## 决策问题

一格混展示金额、方式标签与待付/实付，客服难扫读；与用户押金/用户列表的「方式 vs 状态」口径不一致。

## 调用的大师视角

- **俞军**：拆列后「怎么交的 / 收到没有」替换成本低于解析复合标签。
- **乔布斯**：去掉「实付押金」「待付押」粘连标签，状态独立成列。

## 已确认规则

### 1. 押金方式

| 取值 | 展示 |
|------|------|
| 实付 | `实付` + 副行 `¥应缴额` |
| 信用免押 | `信用免押` + 副行通道与分数 |
| 渠道担保 | `渠道担保` + 副行渠道名（本页少见） |
| —— | 无记录 / 本单应缴 0（**不**展示「无需押金」；decision-068） |

### 2. 收款状态

| 取值 | 含义 |
|------|------|
| 已收 | **仅实付**：`depositPaid ≥ batteryDeposit` |
| 待付 | **仅实付**：未收齐 |
| —— | 信用免押 / 渠道担保 / 无记录 |

> **规则**：非实付不展示「已收/待付」。本列只表达**本单押金收款**，不等于用户台账「在押/退押中」（那是用户押金页）。

## 支持 / 反对 / 失效

- **支持**：与 decision-066 押金拆列一致，降低误读。
- **反对**：列表多一列；宽屏可接受。
- **失效**：产品改回「押金一列含方式+收款」。
