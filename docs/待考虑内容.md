## 5. 开放议题 · 套餐结束后逾期还电（T-47）

> **状态**：**🟣 方案待定**（已记录，未选型，**本期不开发**）  
> **决策卡**：[decision-055](../decisions/decision-055.md)  
> **日期**：2026-07-22  
> **适用范围**：优先 **个人套餐**到期/次数耗尽仍持电池；渠道人天「零额度待还电」见 [decision-049](../decisions/decision-049.md)，**不覆盖逾期计费/罚没**。  
> **说明**：下文 A～F 仅为备选梳理，**不代表推荐结论**；拍板前禁止按任一方案改原型/PRD 验收口径。

### 5.1 问题陈述

| 项 | 内容 |
|----|------|
| **正常预期** | 套餐服务结束（`valid_to` 到达 / 次数耗尽 → `已完结` / `套餐过期` / `次数耗尽`）时，用户**应归还电池** |
| **现实偏差** | 用户可能**逾期多日**才还电（忘记、忙、柜远、故意占用） |
| **现状缺口** | 系统**无逾期状态机、无逾期计费、无罚没/催收闭环**；文档口径写「已完结须已还电」，但未定义「未还电的已完结」 |
| **资产风险** | 电池仍在用户手中；实付押金可作抵押，**信用免押**几乎无经济约束 `[假设：免押用户逾期风险更高]` |

### 5.2 与已有能力的边界

| 已有 | 覆盖 | 不覆盖 |
|------|------|--------|
| decision-049 待还电 | 人天池：无额度/预占失败 → 仅可还电 | 个人套餐到期逾期；逾期天数计费 |
| decision-050/押金退还 | 还电后可退押 | 逾期期间如何处置在押款 |
| 功能结构 §4.6 | 「已完结 · 须已还电」理想态 | 逾期多日才还的异常态 |
| decision-054 | 取消自费兜底 | — |

### 5.3 状态补丁（备选讨论用 · 未选型前不落地）

| 字段/状态 | 备选口径（供讨论） |
|-----------|------|
| `batteryHoldStatus` | `持电中` \| `待还电·服务已结束` \| `已还电` |
| `overdueSince` | 服务结束后仍持电的起始时刻（一般 = `valid_to` 日终或次数耗尽时刻） |
| `overdueDays` | 自然日或 24h 滚动 `[待确认]` |
| 骑手端 | 服务结束后持电 → 首页强提醒「请尽快还电」；扫码**仅还电**（对齐 049 门禁） |
| 运营商 | 「用户」或独立名单：持电逾期列表 + 逾期天数 |
| 押金 | 逾期未还电 → **禁止退押**（已有前置「已还电」；须显式拦截） |

### 5.4 备选方案

#### 方案 A · 仅催还（状态 + 通知，不计费）

**做法**：服务结束仍持电 → 进入「待还电·服务已结束」；短信/站内信 N 次催还；运营商名单跟进；还电后按原规则退押。  
**不计**逾期费、不罚没。

| | |
|--|--|
| ✅ 支持 | 一期成本最低；与 049「仅还电」一致；乔布斯减法 |
| ❌ 反对 | 无经济压力，长尾占用电池难收回；免押用户几乎无约束 |
| ⚠️ 失效 | 逾期率/电池丢失率超运营可接受阈值 |

**适合**：MVP 先闭环「能看见、能催」；把计费留到数据验证后。

---

#### 方案 B · 宽限期 + 逾期日费（从押金/补缴扣）

**做法**：
1. **宽限期** `G` 天（建议 1～3 自然日，`[待确认]`）内免费催还。  
2. 超宽限期按日收 **逾期占用费** `F`（运营商可配，或平台统价下限）。  
3. 扣款顺序：实付在押抵扣 → 不足则拉起补缴 → 达上限（如 ≤ 押金数额或电池残值）停止计费并升级处置。  
4. 还电后：剩余押金可退；已扣逾期费不退。

| | |
|--|--|
| ✅ 支持 | 俞军：新体验=催还有代价，替换成本清晰；约束免押以外用户 |
| ❌ 反对 | 计费/对账/客诉复杂；免押用户仍缺扣款抓手；需法务确认「占用费」表述 |
| ⚠️ 失效 | 客诉激增或监管认定不合理收费 |

**适合**：实付押金占比高、运营要控资产周转时。

---

#### 方案 C · 阶梯押金罚没 / 报失

**做法**（示例阶梯，可配）：

| 逾期天数 | 处置 |
|----------|------|
| 0～G | 仅催还 |
| G+1～D1 | 警告 + 限制新购套餐 |
| D1+1～D2 | 扣罚押金 X% 或固定额 |
| > D2 | 押金全额抵电池损失；工单「电池报失」；账号限制 |

信用免押：走支付分/芝麻违约上报或拉黑新购 `[待确认·通道能力]`。

| | |
|--|--|
| ✅ 支持 | 资产保护最强；与押金产品语义一致（押金担保电池） |
| ❌ 反对 | 体验硬；罚没争议与客服成本高；免押路径需另设计 |
| ⚠️ 失效 | 舆情/投诉率过高，或法务认定格式条款风险 |

**适合**：丢失率高、电池单价高的城市；可与 B 组合（先日费后罚没）。

---

#### 方案 D · 逾期自动续购日租/单次（静默或强提醒后扣款）

**做法**：服务结束后仍持电，系统按日自动购买「1 天畅换」或记「占用日」并从微信/押金扣款，直到还电。

| | |
|--|--|
| ✅ 支持 | 收入连续；用户「等于还在用」 |
| ❌ 反对 | 接近已废止的自费兜底心智（054）；未明示同意有扣款合规风险；张小龙：刺激对抗 |
| ⚠️ 失效 | 用户否认授权、支付通道拒扣、客诉定性「乱扣费」 |

**建议**：**不作为一期首选**；若做须强授权协议 + 显著告知，且与 054 口径书面切割。

---

#### 方案 E · 硬件约束（远程降 SOC / 锁电 · 二期倾向）

**做法**：逾期后 IoT 限制放电或提示「仅可还电柜」。

| | |
|--|--|
| ✅ 支持 | 物理上逼还；减少经济纠纷 |
| ❌ 反对 | 依赖柜机/BMS；安全与路边抛锚风险；一期 IoT 能力未必齐 |
| ⚠️ 失效 | 硬件误伤正常用户或无法覆盖存量柜 |

**建议**：标 **二期**；一期不做。

---

#### 方案 F · 组合拳（备选之一，非结论）

```text
服务结束仍持电
  → 状态：待还电·服务已结束（底座）
  → 0～G 天：仅催还（方案 A）
  → 超 G：日费从押金扣（方案 B）上限 = 押金
  → 超 D2 且未还：报失流程 + 剩余押金结算（方案 C 尾段）
  → 免押用户：催还 + 限制新购 + （二期）信用通道违约
  → 远程锁电：二期（方案 E）
```

| | |
|--|--|
| ✅ 支持 | 宽严有度；可分期开关计费/罚没 |
| ❌ 反对 | 规则条数多，配置与文案成本 |
| ⚠️ 失效 | 运营不愿配参，最终退化成纯催还 |

### 5.5 方案对比（速查 · 非选型）

| 方案 | 一期复杂度 | 资产约束力 | 客诉风险 | 与 049/054 一致性 | 备注 |
|------|------------|------------|----------|-------------------|------|
| A 仅催还 | 低 | 低 | 低 | 高 | 成本最低 |
| B 逾期日费 | 中高 | 中 | 中 | 中 | 需对账与法务 |
| C 阶梯罚没 | 中 | 高 | 高 | 中 | 可与 B 尾段组合 |
| D 自动续购 | 高 | 中 | 高 | 低（近 054） | 慎选 |
| E 远程锁电 | 高（硬） | 高 | 中 | — | 倾向二期 |
| F 组合 | 可分期 | 高 | 可控 | 高 | 备选组合形态 |

### 5.6 异常与边界（补漏）

| 场景 | 处理要点 |
|------|----------|
| 宽限期内续费/新购套餐 | `[待确认]` 是否自动解除逾期、允许继续持电换电 |
| 逾期中申请退押 | 拒绝；文案「请先还电」 |
| 逾期还电但电池损坏 | 走损坏鉴定，与逾期费分开 |
| 次数卡提前耗尽仍持电 | 与按天到期同一套逾期逻辑 |
| 冻结期满自动解冻 vs 到期 | 冻结前须还电（已有）；本议题聚焦**到期未还** |
| 渠道人天 vs 个人 | 人天用 049；个人用本议题；**勿混用计费池** |
| 运营商配置权 | 宽限天数、日费、罚没阶梯是否运营商可配还是平台统管 `[待确认]` |
