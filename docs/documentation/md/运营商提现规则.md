# 运营商提现规则

> C 端支付**实时清分**入账运营商子商户；**提现**须运营商申请、平台审核通过后打款至绑定账户。  
> **仅适用于运营商**；**渠道商-设备租赁**不适用（白名单套餐款进渠道子商户，不经本流程）。

关联：[合作模式与分账.md](./合作模式与分账.md) §3.2 · [PRD.md](./PRD.md) §5.14 · [融资管理-PRD.md](./融资管理-PRD.md)

---

## 1. 清分与提现分离

| 环节 | 时点 | 说明 |
|------|------|------|
| **清分** | 支付成功即时 | 平台 1% + 运营商 99% 进入运营商子商户，状态 `已清分` |
| **提现** | 运营商申请 + 平台审核 | 从可提现余额划出至运营商绑定结算账户 |

清分不等于自动出款；运营商子商户内资金须走提现申请。

---

## 2. 可提现余额

```
可提现余额 = max(0, 已清分余额 − 已提现 − 待审/处理中提现 − 本月融资待还)
```

| 项 | 说明 |
|----|------|
| **已清分余额** | 支付成功实时清分、尚未提现的运营商净额累计 |
| **已提现** | 状态为「已提现」的申请金额合计 |
| **待审/处理中** | 「待审核」「审核通过」「处理中」申请金额合计 |
| **本月融资待还** | 融资管理 → 还款日历中，**当月**状态非「已还清」的 `dueAmount − paidAmount` 合计 |

融资待还为**预留扣减**，防止运营商在待还到期前过度提现；还款工单确认后相应释放。

---

## 3. 提现流程

```mermaid
flowchart LR
  A[运营商发起申请] --> B[平台审核]
  B -->|通过| C[发起提现/代付]
  B -->|驳回| D[结束·可重新申请]
  C --> E[到账运营商绑定账户]
```

| 步骤 | 操作方 | 说明 |
|------|--------|------|
| 1 | 运营商 | 「我的流水 → 提现申请」填写金额；不得超过可提现余额 |
| 2 | 运营商 | 选择默认收款子商户对应结算卡（见「收款账户」） |
| 3 | 平台 | 「流水管理 → 运营商提现审核」审核 |
| 4 | 平台 | 通过后发起通道提现/代付 |
| 5 | 系统 | 状态更新为「已提现」，记录到账时间 |

**驳回**：须填写原因；运营商可修正后重新申请。

---

## 4. 适用范围

| 主体 | 是否适用 | 说明 |
|------|----------|------|
| **运营商** | ✓ | 骑手套餐 C 端收款子商户 |
| **渠道商-人天池** | — | B2B 采购款在采购时结清，无本提现流 |
| **渠道商-设备租赁** | ✗ | 白名单套餐款进**渠道**子商户，不经运营商提现审核 |
| **渠道商-骑士卡** | — | 佣金及时到付进渠道子商户（若开启） |
| **平台** | — | 平台自有商户独立结算 |

---

## 5. 字段建议

| 实体 | 字段 |
|------|------|
| `WithdrawalRequest` | `id`, `operator_id`, `amount`, `account_id`, `status`, `apply_time`, `review_time`, `reviewed_by`, `withdraw_time`, `month_due_reserved`, `reject_reason` |
| `status` | `待审核` / `审核通过` / `处理中` / `已提现` / `已驳回` |

---

## 6. 验收要点

| # | 场景 | 预期 |
|---|------|------|
| 1 | 支付成功 | 清分明细 `已清分`；总览已清分增加 |
| 2 | 有融资待还 | 可提现 = 已清分 − 已提现 − 待审 − 本月待还；演示 OP-SX 扣 ¥10,500 |
| 3 | 发起申请 | 生成待审核单；待审金额占用可提现 |
| 4 | 平台通过 | 状态已提现；记录到账账户与时间 |
| 5 | 平台驳回 | 状态已驳回；释放待审占用 |
| 6 | 设备租赁渠道 | 收款账户页注明不适用运营商提现审核 |
