# 税收 · 待讨论

> **状态**：待财务/法务讨论 · **不纳入一期产品与原型**  
> 关联：[税筹与分账结构参考方案.md](./税筹与分账结构参考方案.md) · [合作模式与分账.md](./合作模式与分账.md) §8.2

---

## 1. 定位

一期产品按 **架构 B** 落地：C 端套餐款进入**运营商单一进件子商户**，支付成功实时清分，平台 1% 分账。

本节讨论「**套餐收入如何在合规前提下拆分为多个成本/服务基础**」，以降低运营商账面营收、优化税筹结构。**讨论结论未拍板前，后台不配置多收款主体、不合单、不做 SKU 法人拆分。**

---

## 2. 动机（会议背景）

若 C 端 300 元套餐全额计入运营商营收，易快速突破**小规模纳税人 500 万**门槛，且进项抵扣受限，综合税负上升。

目标：在具备**业务实质**的前提下，将用户支付金额归属到不同服务提供方或成本项，各主体按归属记账、开票。

---

## 3. 服务拆分基础（讨论框架）

用户支付的一笔套餐费，理论上可拆为以下 **服务基础**（非一期实现，供财务建模）：

| 基础 | 范围 | 典型归属主体（待议） | 说明 |
|------|------|----------------------|------|
| **设备** | 换电柜、电池 | 运营商 / 租赁 SPV / 资方 | 设备折旧、租赁费、保修 |
| **场地** | 站点租金、物业 | 场地方 / 运营商区域公司 | 与站点合伙人模型可衔接 |
| **电费** | 柜机用电、补能 | 能源公司 / 运营商 | 可计量或按次分摊 |
| **平台** | 软件 SaaS、设备运维 | 平台公司 | 与现有 1% 技术服务费区分建模 |

> **原则**：拆分须有真实服务或合同支撑，不能仅为降税而虚构交易（参见原「业务实质」评审要求）。

### 3.1 与「多法人主体」方案的关系

曾探索的 **架构 B+**（同一运营品牌下多法人、合单支付、SKU 按法人拆分子订单）已**撤出产品与原型**，其设计要点归档如下，供税收讨论参考：

| 曾探索点 | 说明 |
|----------|------|
| 合单支付 | 骑手一次支付，款项进入多个子商户 |
| SKU 拆分行 | 如「换电服务费」+「电池使用费」进不同法人 |
| 区域路由 | 按购卡站点/城市解析区域法人 |
| 三一致 | 协议销售方 = 收款子商户 = 发票开具方 |

**当前结论**：上述能力**不开发**；若财务确定拆分方案，再评估是「记账拆分」还是「多主体收款」。

## 4. 与架构 B（一期）的边界

| 维度 | 一期（架构 B） | 税收讨论态 |
|------|----------------|------------|
| C 端收款 | 运营商**单**子商户 | 可能多主体或监管户 |
| 后台菜单 | 收款账户、资金实收 | **无**法人主体/拆分规则/合单 |
| 骑手购卡 | 单一收银台、单一协议 | 可能复合协议、分项展示 |
| 退款 | 运营商子商户原路退 | 若多主体则分子单退 |
| userOwner | operator_id | **不变** |
