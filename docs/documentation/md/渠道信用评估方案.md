> **状态**：已确认（2026-06-17）  
> **决策来源**：[需求评审决策记录.md](./需求评审决策记录.md) #6、#11、C-01–C-05  
> **关联**：D16 渠道信用额度、D38 渠道用户免押、D35 卢俊团队历史方案  
> **一期边界**：**评估方案一期交付**（模型、分级、后台、状态机）；外部征信/第三方数据可 Mock，规则须闭合

---

## 1. 目标与边界

### 1.1 要解决什么

渠道签约骑手的**电池/柜机押金**不落在骑手个人，统一算在**渠道商主体**。平台评估渠道信用并告知运营商；运营商可在平台评估基础上**调整信用额度**。

| 信用档位 | 渠道侧义务 |
|----------|------------|
| **按评分分级** | 信用抵扣应押总额；不足部分渠道线下补押 |

### 1.2 不解决什么

- 不替代 C 端个人用户的芝麻信用 / 微信支付分免押
- **渠道分销**（推广链接直购，如骑士卡）：用户按个人套餐押金/免押规则，**不适用**本文渠道信用额度
- 不自动从渠道批发款中扣押金（押金与采购款分账）
- 一期**不阻断**换电（押金不足仅预警）

### 1.3 角色分工

| 角色 | 职责 |
|------|------|
| **平台管理员** | **统一设置**设备押金标准（电池/换电柜）；维护信用评估模型与**分级抵扣规则**（可配置）；新签渠道**手工初评**；向运营商输出评估结果 |
| **运营商** | 查看平台评估；**可修改渠道信用额度**（调高/调低，留审计）；接收押金缺口提醒；**审核**渠道补缴凭证 |
| **渠道商** | 查看信用与应押/抵扣/缺口；**提交线下打款凭证**待运营商审核 |

## 2. 押金标准（C-01 · 平台统一设置）

| 设备类型 | 平台默认值 | 维护方 |
|----------|------------|--------|
| **电池** | **¥3,000 / 块** | 平台管理员（全平台统一） |
| **换电柜** | **¥15,000 / 台** | 平台管理员（全平台统一） |

**渠道骑手**：默认**免押**（不对骑手个人收取）；押金责任**统一算在渠道商**。

### 2.1 应押总额公式

```
应押总额 = 在服渠道骑手数 × 单骑手应押标准
单骑手应押标准 = 平台统一配置（默认按「1 电池」计 ¥3,000；若合同含柜机分摊则按签约约定）
信用抵扣额 = min(应押总额, 渠道有效信用额度)
渠道待补缴 = max(0, 应押总额 − 信用抵扣额)
```

| 字段 | 维护方 | 说明 |
|------|--------|------|
| `batteryDepositStandard` | **平台管理员** | 全平台统一，默认 ¥3,000 |
| `cabinetDepositStandard` | **平台管理员** | 全平台统一，默认 ¥15,000 |
| `activeChannelRiders` | 系统 | 有效渠道成员数 |
| `creditLimit` | 平台评估 → **运营商可改** | 渠道有效信用额度（元） |
| `creditScore` | 平台评估 | 信用评分（0–100） |

## 3. 评估体系

### 3.1 评估维度（建议权重 · 可配置）

| 维度 | 权重建议 | 数据来源 | 一期 |
|------|----------|----------|------|
| **合作年限** | 15% | 平台签约日期 | ✅ |
| **累计采购额** | 20% | B2B 批发/购卡/租金池充值 | ✅（平台内数据） |
| **消耗履约率** | 25% | 预占成功率、离职回池率、欠费天数 | ✅ |
| **押金历史** | 20% | 补缴记录、逾期天数 | ✅ |
| **主体资质** | 10% | 企业类型、注册资本 | 手工录入 |
| **第三方信用** | 10% | 企业征信 / 行为分 | Mock（二期可接） |

### 3.2 信用评分 → 分级抵扣（C-02 · 可配置）

平台管理员在后台维护**评分区间与抵扣比例**（不写死，支持自行设置）：

| 等级（示例） | 得分区间 | 抵扣比例（示例 · 可改） |
|--------------|----------|-------------------------|
| 优 | ≥ 85 | 100% |
| 良 | 70–84 | 60% |
| 中 | 55–69 | 30% |
| 差 | < 55 | 0% |

```
渠道信用额度 = 应押总额 × 对应等级抵扣比例
```

运营商可在平台给出额度基础上**手动调整** `creditLimit`（须填原因，留 `ChannelCreditEvalLog`）。

### 3.3 初评（C-03）

| 规则 | 说明 |
|------|------|
| **新签渠道** | **不设默认等级**；由**平台管理员**签约时手工定级并录入初评分数 |
| **复评** | 每月 1 日自动复评；采购额突变 / 骑手规模大幅变动时触发 |
| **申诉** | 运营商可向平台申请复核 |

## 4. 不足提醒与补缴（C-04）

| 步骤 | 角色 | 动作 |
|------|------|------|
| 1 | 系统 | `待补缴 > 0` → 通知运营商（+ 抄送渠道） |
| 2 | **渠道商** | 线下打款后，在「渠道信用额度」**提交打款凭证**（金额、流水号、转账日、附件） |
| 3 | **运营商** | **审核**凭证：通过 → 销账；驳回 → 渠道重新提交 |
| 4 | 系统 | 一期**不阻断**换电 |

| 场景 | 系统行为 |
|------|----------|
| 连续 30 日未补缴 | 升级提醒平台管理员（建议） |
| 运营商配置「押金不足停服」 | **二期** |

## 6. 后台功能

| 页面 | 角色 | 能力 |
|------|------|------|
| **平台 · 押金标准** | 平台管理员 | 编辑电池 ¥3,000、换电柜 ¥15,000 等全平台默认值 |
| **平台 · 信用规则** | 平台管理员 | 配置评分维度权重、等级区间、抵扣比例 |
| **平台 · 渠道商管理** | 平台管理员 | 新签初评、复评触发、人工覆写 |
| **渠道销售 → 签约详情** | 运营商 | 只读平台评估；**修改信用额度** |
| **渠道信用额度** | 渠道商 | 信用分/等级/额度/应押/缺口；**提交打款凭证** |
| **运营商总览** | 运营商 | 渠道押金缺口 KPI；待审核凭证列表 |
