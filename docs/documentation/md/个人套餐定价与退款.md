## 1. 计费维度：按天 vs 按次

| 维度 | 含义 | 典型权益 | 收入确认 |
|------|------|----------|----------|
| **按天** | 购买后 N 自然日或 24h 内**不限次换电** | 7天 / 30天畅换 / 90天季卡 / 1天畅换 | 按日摊销 `P/D` |
| **按次** | 购买后有效期内**消耗换电次数** | 单次 / 10次卡 / 30次卡 | 按次或首换当笔确认 |

> **已确认**：1 天与单次均为 **购买时刻起 24 小时内有效**（非自然日 0:00 切日）。

### 生效时间规则（方案 B）

方案 B 采用 **两套计时口径**，开发须按 SKU 类型分支：

| 计时口径 | 适用 SKU | 起算点 | 截止点 |
|----------|----------|--------|--------|
| **滚动 24 小时** | 1天畅换、单次换电 | `pay_time`（支付成功时刻） | `pay_time + 24h` |
| **自然日（含购买日）** | 7天、30天畅换、90天季卡、10次卡、30次卡 | `pay_date`（支付成功当日） | 见下表 |

**自然日型 `valid_from` / `valid_to` 计算**：

```
valid_from = date(pay_time)                    // 购买当日为第 1 天
valid_to   = valid_from + (validity_days - 1)  // 含首尾两个自然日
服务截止   = valid_to 当日 23:59:59（运营商时区，默认 Asia/Shanghai）
```

| SKU | validity_days | 示例（pay_time = 2026-05-01 09:12） |
|------|---------------|-------------------------------------|
| 7天套餐 | 7 | valid_to = 2026-05-07 |
| 30天畅换 | 30 | valid_to = 2026-05-30 |
| 90天季卡 | 90 | valid_to = 2026-07-29 |
| 10次卡 | 30（次数窗口） | valid_to = 2026-05-30；10 次用完或到期，以先到为准 |
| 30次卡 | 60（次数窗口） | valid_to = 2026-06-29 |

**滚动 24h 型示例**（pay_time = 2026-06-09 07:50）：

| SKU | 截止 |
|-----|------|
| 1天畅换 | 2026-06-10 07:50 前不限次 |
| 单次换电 | 2026-06-10 07:50 前可换 1 次；换电成功即完结 |

**权益校验**（换电扫码 / 渠道骑手可换电校验）：

```
allowed = now ∈ [valid_from, 服务截止]
          AND service_state = 服务中
          AND (按次卡：swap_used < swap_limit)
```

**与其他状态的关系**：

| 场景 | 生效时间处理 |
|------|--------------|
| **服务冻结** | 冻结期间计时**暂停**；解冻后 `valid_to` **顺延**冻结天数（见 PRD §5.2） |
| **中途完结退款** | 立即终止；剩余按 §5.2 计算 |
| **次卡过期未用完** | 到期自动完结，剩余次数作废（不退） |
| **跨站换电** | 不改变已购套餐的 `valid_from/valid_to`，仅校验是否在有效期内 |

**定价配置生效**（运营商改价）：

| 层级 | 规则 |
|------|------|
| 通用 / 默认 / 站点覆盖价 | 保存后立即对新购订单生效；**已购套餐**沿用 `price_snapshot` |
| 下架某 SKU | 禁止新购；已购订单按原服务期履约 |

> **与 7天/30天/90天的区别**：默认套餐的「1天」是 **24h 滚动**，不是「自然日 1 天」；避免与 7天套餐（自然日）混淆。骑手端须明确展示截止时刻（如「今日 07:50 前有效」vs「至 05-30」）。

### 收入确认摘要

| SKU | 确认方式 |
|-----|----------|
| 7天 / 30天畅换 / 90天季卡 | 按日摊销 |
| 1天畅换 | 未换电可全额退；已换电按日或当笔（与配置一致） |
| 10次卡 / 30次卡 | 按次确认或换电当笔 |
| 单次 | 换电成功当笔确认 |

## 2. 定价架构（通用 / 新人 / 默认 / 站点）

### 2.1 层级定义

```
运营商
├── 通用套餐（priceScope=general）
│   └── 面向个人用户的常规 SKU（7天/30天畅换/季卡/次卡等）
├── 通用新人套餐（priceScope=newcomer_general）
│   └── 运营商级新人专享价；用户首购时可享
├── 默认套餐（priceScope=fallback，渠道应急）
│   └── 必选上架：1天畅换 + 单次；渠道无预占时引导在线支付
├── 站点套餐（priceScope=site）
│   └── 覆盖「通用套餐」零售价；未配置则继承通用
└── 站点新人套餐（priceScope=newcomer_site）
    └── 覆盖「通用新人套餐」；未配置则首购继承通用新人
```

### 2.2 新人套餐规则（已确认）

| 规则 | 说明 |
|------|------|
| **适用对象** | 个人套餐首购用户（非渠道应急兜底） |
| **资格判定** | `user_id` × `operator_id`（售卖运营商）维度，历史上**无支付成功**的个人 `PackageOrder` |
| **使用次数** | **每用户每运营商仅 1 次**；支付成功即消耗资格 |
| **退款** | 中途退款**不恢复**新人资格 |
| **渠道成员** | 走渠道人天/应急兜底时不适用新人价；若以个人身份首购则适用 |
| **互斥** | 新人价与常规价**二选一**展示，首购优先新人货架 |

**首购价格解析顺序**（用户在站点 S 下单）：

```
1. 站点新人套餐（newcomer_site @ S）— 若配置且生效
2. 通用新人套餐（newcomer_general）— 若配置且生效
3. 常规套餐价（站点覆盖 > 通用）
```

### 2.3 定价维度与继承

```
运营商 × 城市 × priceScope × SKU → 零售价
站点覆盖：运营商 × 城市 × 站点 × SKU → 零售价（仅通用类 SKU）
```

**继承优先级**：

| 类型 | 优先级 |
|------|--------|
| 常规零售价 | 站点覆盖 > 通用套餐 |
| 新人首购价 | 站点新人 > 通用新人 |

| 层级 | 站点字段 | 新人标记 | 可删除 |
|------|----------|----------|--------|
| 通用套餐 | 空（全站点） | 否 | 是 |
| 通用新人套餐 | 空（全站点） | **是** | 是 |
| 默认套餐 | 空（全站点） | 否（渠道应急） | **否** |
| 站点套餐 | 具体站点 | 否 | 是（恢复继承） |
| 站点新人套餐 | 具体站点 | **是** | 是（恢复继承通用新人） |

### 2.4 站点生效规则

| 场景 | 生效价来源 |
|------|------------|
| 站点未配置某 SKU | 继承**通用套餐**同 SKU 价 |
| 站点已配置常规覆盖 | 使用**站点套餐**价（非首购或已无新人资格） |
| 首购且无站点新人 | 使用**通用新人套餐**（若有） |
| 首购且有站点新人 | 使用**站点新人套餐** |
| 渠道成员无预占兜底 | 始终使用**默认套餐**（运营商级） |
| 跨站换电 | 价格锁定在**购买套餐单**快照，不因换电站变化 |

### 2.5 是否必须分站点不同价

| 场景 | 建议 |
|------|------|
| 同城统一定价 | 仅维护通用套餐即可 |
| 站点场地成本不同 | 为部分 SKU 设站点覆盖价（与合伙人分佣无关） |
| 新站点开业促销 | 临时站点覆盖，活动结束后删除恢复继承 |

## 3. 支付路径

### 3.1 个人用户

选择当前站点可用 SKU（按继承规则解析）→ **在线支付** → 平台 **C 端 1%** 分账 → 生成 `PackageOrder`。

### 3.2 渠道成员 · 无预占人天

当日**未预占 / 预占失败 / 无分配额度**时：

1. 换电校验失败，引导**在线支付**；
2. **默认推荐 1天畅换或单次 SKU**（**已确认 Q1**），不默认推 30天畅换/季卡等长周期 SKU；
3. 支付成功生成 `PackageOrder`，`source=channel_retail_fallback`；
4. 平台 **C 端 1%**（支付成功时），与 personal 相同；
5. 渠道兜底 C 端支付按 **平台 1% + 运营商 99%** 清分。

### 3.3 默认套餐（运营商强制）

**已确认（Q4）**：面向渠道用户应急，运营商在定价管理中**必须**维护且保持上架：

| 必选 SKU | 用途 |
|----------|------|
| **1 天畅换套餐** | 渠道骑手无预占时的默认兜底之一 |
| **单次套餐** | 渠道骑手无预占时的默认兜底之一 |

- 归属 **默认套餐** 层级，至少各 1 个运营商级生效价格；
- 下架或缺失时，系统**阻止**该运营商渠道成员进入「自费兜底」路径，并提示运营商补配置；
- 与人天批发价独立；兜底零售价为 C 端在线支付价。

## 4.5 开通冷静期（3 天）

> **已确认**：用户在开通服务后 **3 个自然日内**可申请退**大部分费用**（扣除已使用天数）；**默认须运营商审核**，具体退款金额由运营商在确认时决定。超过 3 天则**不再享受冷静期强制退款**，平台**不主动退还**。

### 4.5.1 起算与窗口

| 项 | 规则 |
|----|------|
| 起算点 | `pay_time`（支付成功时刻） |
| 窗口 | 起算后 **3 个自然日**（含支付当日） |
| 超过窗口 | 冷静期入口关闭；可走 §5 中途完结 / SKU 未使用退订 |

### 4.5.2 准入（C 端申请）

| # | 条件 | 不满足 |
|---|------|--------|
| 1 | 在冷静期窗口内 | 「已超过 3 天冷静期，不可申请」 |
| 2 | 状态 = **服务中** | 拒绝 |
| 3 | **未持有电池** | 「请先归还电池」 |
| 4 | `source ≠ day_pool` | 渠道池订单不适用 |

### 4.5.3 建议应退金额（系统计算，运营商可改）

**天数型套餐**（7天 / 30天畅换 / 90天季卡）：

```
建议套餐退 R = round( P × remaining_days / total_days , 2 )
```

- `remaining_days`：自申请日起至 `valid_to` 的剩余自然日（与 §5.2 一致）
- `used_days = total_days - remaining_days`（用于展示「已使用天数」）
- **押金**：已还电且实付押金 → 建议全额退押金；免押为 0

> 运营商在后台「确认退款」时可修改**实退套餐费**与**押金退**，系统记录调整说明。

### 4.5.4 审核与执行

| 项 | 规则 |
|----|------|
| 审核 | **默认须运营商审核**；不受「自动退款」模式开关影响 |
| 执行 | 运营商确认后 → 终止 `PackageOrder` → **运营商子商户原路退** → 清分冲正 |
| 超过冷静期 | 平台不主动退；用户仅可按 §5.1 门槛申请中途完结等 |

## 5. 中途退款（自动）

### 5.1 准入（系统自动校验）

| # | 条件 | 不满足 |
|---|------|--------|
| 1 | 状态 = **服务中** | 拒绝 |
| 2 | **未持有电池** | 「请先归还电池」 |
| 3 | **剩余服务 ≥ 1 单位** | 「剩余不足 1 天/1 次，不可退款」 |
| 4 | `source ≠ day_pool` | 渠道池订单不退骑手（离职回池） |

**剩余单位**：

| SKU | 计算 |
|-----|------|
| 7天 / 30天畅换 / 90天季卡 | 剩余自然日 ≥ 1 天（`valid_to` 当日 23:59:59 前）；**不足 1 自然日不可退** |
| 1天畅换（24h） | 剩余有效时长 ≥ 24h 视为 ≥ 1 单位；**不足 24h 不可退** |
| 单次 | 未换电（剩余 1 次）可退；已换电不可退 |
| 10次卡 / 30次卡 | 剩余次数 ≥ 1 且未过期 |

### 5.2 应退金额

**天数型（7天 / 30天畅换 / 90天季卡）**：

```
R = round( P × remaining_days / total_days , 2 )
```

**1天畅换（24h）**：未换电 → `R = P`；已换电 → `R = 0`。

**单次（24h）**：未换电 → `R = P`；已换电 → `R = 0`。

**10次卡 / 30次卡**：

```
R = round( P × remaining_swaps / total_swaps , 2 )
```

### 5.3 退款执行流程

运营商在后台「退款管理」可配置：

| 模式 | 行为 |
|------|------|
| **自动退款**（默认推荐） | 符合条件 → **无需人工审核** → 终止 `PackageOrder` → **运营商子商户原路退** `R` → 清分/收入冲正 |
| **手动确认** | 申请进入待审核；运营商确认后**系统自动执行**运营商原路退/垫付（非线下手工打款） |

详见 [PRD.md](./PRD.md) §5.2.2。

### 5.4 退款清分冲正

| 部分 | 处理 |
|------|------|
| 清分台账 | **运营商子商户**出款原路退 R；平台 1% **不退**；生成负向清分明细 |
| 未确认预收（收入侧） | 直接冲减，停止后续摊销 |
| 已确认部分（收入侧） | 按已摊销比例冲减 `recognized_amount` |
| 已提现 | 可退余额不足 → 标记**须垫付** |

**渠道自费兜底订单**：仅平台 1% + 运营商。
